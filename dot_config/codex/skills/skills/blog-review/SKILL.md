# ブログレビュー

## 概要

ブログ記事の品質をレビューし、優先度付きで改善提案を提示する。

## ワークフロー

1. 目的と前提を確認する。
2. 記事本文を読み、要約と読者体験を整理する。
3. 重要度順に指摘と提案をまとめる。
4. 具体的な改善例を提示する。

## 1. 目的と前提を確認する

不足情報があれば短く質問する。

- 想定読者、目的、媒体、トーン
- 文字数目標、締切、公開先の制約
- 既存のルール（表記、禁則、リンク方針）

## 2. レビュー観点

以下を確認し、影響の大きい順に整理する。

- 構成: タイトル、導入、見出し階層、結論の明確さ
- 論理: 主張と根拠の対応、段落のつながり
- 可読性: 一文の長さ、指示語の曖昧さ、冗長表現
- トーン: 読者に対する敬語、距離感、語調の一貫性
- 具体性: 例・データ・手順の不足
- 正確性: 事実関係、用語の誤用
- 新規性: 既存情報との差別化、独自の視点
- 炎上リスク: 誤解を生む表現、断定の強さ、配慮不足

## 3. 出力形式

必ず以下の順で出力し、見出しと区切りを保つ。

1. 目的・前提（不足時のみ質問）
2. 要約（2〜4文）
3. 観点別スコア表（5段階）
4. 指摘一覧（重大度別）
5. 改善例（必要箇所のみ）

### 観点別スコア表の書式

```
観点別スコア（5段階）
- 構成: 3/5
- 論理: 2/5
- 可読性: 4/5
- トーン: 3/5
- 具体性: 2/5
- 正確性: 4/5
- 新規性: 2/5
- 炎上リスク: 3/5
```

### 指摘一覧の書式

```
指摘一覧
【重大度: 高】
1) 観点: 論理
   問題点: ...
   影響: ...
   提案: ...

【重大度: 中】
1) 観点: 可読性
   問題点: ...
   影響: ...
   提案: ...

【重大度: 低】
1) 観点: トーン
   問題点: ...
   影響: ...
   提案: ...
```

指摘は「問題点 → 影響 → 提案」の順で書く。
指摘には必ず対象の観点名を明記する。

## 4. 改善例の作り方

- 必要最小限の範囲だけを書き換える
- 変更理由を一言添える
- 全面リライトは依頼がある場合のみ行う

## 5. 例

入力例:
「この記事をレビューしてください。想定読者は初学者です。」

出力例（形式のみ）:
- 要約: ...
- 重大度高: ...
- 中程度: ...
- 低優先: ...
- 改善例: ...
